拓扑排序

把一个图，排序成一个线性序列。
方法：类似于广度优先

  1. 建立图和入度表
  2. 先把入度为0的起始点加入队列
  3. 如果队列不为空，弹出队列中的第一个点，加入res
  4. 将与这个点相连的点的入度减一，如果为0，则加入队列
  5. 执行2，3直到队列为空
  6. 如果res的长度和点数相同，说明排序完成，否则说明图中有环

dijkstra 单源最短路径算法

计算从起点出发，到每个点的最短距离
算法思路：
每次从”未求出最短路径的点”中取出“距离起点最近的点”，以这个点为桥梁刷新“未求出最短路径的点的”距离
也就是记录每个点到起点的最短距离
流程：

  1. 建立临接链表edges[i] = append(edges[i], edge{city, cost})
    建立初始dist[i] = math.MaxInt32
  2. 把起点放进优先队列
  3. 对于优先队列中的每一个点执行以下循环：
  • 弹出cost最小的加入结果队列，cost为d1
  • 遍历该点的所有临接，如果d1+v.cost<dist[i], 说明之前没有访问过，更新dist[i]和v.cost
  1. 如果len(结果队列)==点的总数，返回结果队列最后一个点的cost, 否则返回-1

ACID of transaction

Posted on 2022-09-25 Edited on 2022-09-26

数据库的逻辑架构分为三层：

  1. 处理连接
  2. 缓存，解析和优化语句
  3. 存储 和 提取数据

事务不是数据库服务器层实现的，而是存储引擎实现的
InnoDB引擎支持事务

语法：

四个特性：

Atomicity 原子性
consistency 一致性
Isolation 隔离性
durability 持久性

事务不一定满足这四个性质
如innodb默认隔离级别是：可重复读RR， 不满足隔离性

实现：

for 原子性：undo log

原子性就是保证全部执行，如果出错，则全部rollback
所以rollback是关键， undolog 就是用来rollback的
undolog是逻辑日志，记录执行操作，之前的值，之后的值，rollback就是根据undolog执行相反的语句以恢复数据

for 持久性：redo log

持久性是保证commit之后，对数据库的改变是永久的。而不能受到故障的影响。
数据 –> buffer pool –> 磁盘
buffer pool 中存放的是磁盘中部分数据页的映射。
持久性问题主要是由于buffer pool中数据未能落盘导致的。

buffer pool 是每隔一段时间，以页(16kb)为单位刷盘

改进方法：
数据修改时，先写redolog，再写buffer pool, 当commit的时候，调用redo log对数据进行刷盘。这样如果数据库宕机，在重启的时候就可以根据redo log进行恢复。

redolog刷盘和buffer pool刷盘之间的区别：

  1. redolog是顺序io, buffer pool是随机io
  2. redolog只追加修改部分，buffer pool要刷修改部分所在的整个一页数据
    综上，redolog比buffer pool要更快更少，效率更高一些。

binlog和redolog的区别：
binlog是二进制，记录执行了什么语句的执行日志，在事务commit的时候写入，定期刷盘。主要用来根据时间点恢复数据，和主从复制。

for 一致性

一致性是最终的目标，由前三者共同保障，不仅需要数据库层面的保障，还需要应用层的保障

for 隔离性: undo log

隔离性是隔离不同事务，使 “并发执行” 的事务之间相互不干扰。
真正的隔离性级别是：serialization
假设有A,B两个事务，以下4个场景：
A读B写： MVCC
A写B读
A读B读
A写B写： 锁

锁分为行锁，表锁
锁分为读锁（共享锁），写锁（排它锁）

读操作存在的问题：

  1. 脏读：B事务修改数据未提交，A读到B未提交的数据
  2. 不可重复读：B事务修改数据已提交，A在B提交前读的数据，和B提交后分别读取一次数据，两次结果不一样
  3. 幻读：也是不可重复读，但区别是B不是修改数据，而是新增数据，A查询到的数据行数变了。

事务的隔离级别：
读未提交：啥问题都有
读已提交：解决脏读
可重复读：解决不可重复读
串行化：解决幻读

InnoDB实现的RR避免了幻读问题 ？ 那不就是实现了隔离性？
MVCC最大的优点是，读不加锁，所以读写不冲突，并发性能好。

读分为两种：快照读，当前读
mvcc不加锁：就是快照读
加锁：就是当前读
什么情况下是加锁：
共享锁读取
select…lock in share mode
排它锁读取
select…for update
update …
如果先快照读，再当前读，就会发生数据不一致的情况
还可以通过next-key lock(record lock行锁 + gap lock间隙锁) 来解决幻读的问题

认识自己的方法

理论：

  • 约哈里窗
    方式：
  • 他人视角：“认知多维镜”
  • 自我观察：“局外人”视角

    约哈里窗

    “自我意识的发现-反馈模型”
    四个窗口：
    自己知道，自己未知
    他人知道，他人未知

自己知道，他人知道：开放我 - 外显个性
自己知道，他人未知：隐藏我 - 个人癖好，童年记忆，痛苦经验
自己未知，他人知道：盲目我 - 优缺点，习惯性动作，某些对人处事的态度
自己未知，他人未知：未知我 - 潜能，天赋

我们应该将开放我最大化，这样能够对自我认识更加透彻，也能增进他人对自己的理解，让自己内外一致，活的更加坦荡，更容易被人接受。

我们要尽可能的展示自己的优势，同时不要惧怕暴露自己的一些问题。只有人们展示信息足够多和充分时，人们才能接受到来自他人准确的反馈。

我们遇见所有人，都是为了遇见自己，一个更完整的自己。

避免和外界过多的接触，开放我就会小，不认识自己，就会产生内外冲突感。
所以要更加主动的尝试一些方法：
分享自己，倾听他人，展开讨论

方法：

  1. 开放自我，坦诚交友 - 收集反馈，减少盲目我
  2. 建立信任 - 开放隐藏我

通过自我坦诚与反馈，逐步缩小盲目我和隐藏我，使开放我更加广阔。
在他人说话的时候不要打断，也不要急于解释，认真倾听才能得到更多有价值的反馈。

AE在wrk中的应用

wrk开启压测的流程

主要参数：线程数，连接数，压测时间，超时时间，请求头，压测脚本
主要对象：cfg, sock, statistics, http_parse_settings
线程对象属性：eventloop, 连接数，lua虚拟机

  1. 解析参数
  2. 定义 schema, host, port, service
  3. 如果schema解析为https，进行ssl连接
  4. 设置忽略 SIGINT（interrupt 中断信号）, SIGPIPE （管道破裂信号）， 否则会导致压测服务退出
  5. 根据压测脚本初始化全局的lua虚拟机
  6. 进行地址解析和socket连接，测试host是否可连接
  7. 根据设置的线程数，创建线程对象，执行线程任务
  8. 睡眠等待压测时间
  9. 阻塞等待所有线程结束
  10. 统计结果输出

每个线程执行的任务

  1. 根据每个线程需要启动的连接数，建立socket连接
  2. 根据eventloop建立定时事件，每100ms统计一次数据
  3. 开始处理eventloop
  4. 删除eventloop

建立socket连接做了什么

  1. 建立fd
  2. 设置为O_NONBLOCK
  3. 开启连接
  4. 创建一个文件事件

文件事件中做了什么

  1. 查看socket连接状态
  2. 初始化httpparser
  3. 创建两个文件事件，fd可读事件，fd可写事件

fd可写事件

超时或者写完，删除可写事件

fd可读事件

执行http_parser_execute发送请求，等待结果返回，读响应结果，回调response函数
