  2. 定义 schema, host, port, service
  3. 如果schema解析为https，进行ssl连接
  4. 设置忽略 SIGINT（interrupt 中断信号）, SIGPIPE （管道破裂信号）， 否则会导致压测服务退出
  5. 根据压测脚本初始化全局的lua虚拟机
  6. 进行地址解析和socket连接，测试host是否可连接
  7. 根据设置的线程数，创建线程对象，执行线程任务
  8. 睡眠等待压测时间
  9. 阻塞等待所有线程结束
  10. 统计结果输出

每个线程执行的任务

  1. 根据每个线程需要启动的连接数，建立socket连接
  2. 根据eventloop建立定时事件，每100ms统计一次数据
  3. 开始处理eventloop
  4. 删除eventloop

建立socket连接做了什么

  1. 建立fd
  2. 设置为O_NONBLOCK
  3. 开启连接
  4. 创建一个文件事件

文件事件中做了什么

  1. 查看socket连接状态
  2. 初始化httpparser
  3. 创建两个文件事件，fd可读事件，fd可写事件

http request response parser for c

HTTP Parser
这是一个用C编写的HTTP消息的解析器，它同时解析请求和响应。解析器设计用于性能HTTP应用程序。它不进行任何系统调用或分配，也不缓冲数据，可以随时中断。根据您的体系结构，每个消息流只需要大约40字节的数据（在每个连接的web服务器中）。

Features:

No dependencies
处理持久流(keep-alive)。
解码分块编码。
Upgrade support
防御缓冲区溢出攻击。
解析器从HTTP消息中提取以下信息：

标题字段和值
Content-Length
Request method
响应状态代码
Transfer-Encoding
HTTP version
Request URL
Message body
Usage
每个TCP连接使用一个http_parser对象。使用http_parser_init()初始化结构并设置回调。对于请求解析器，这可能类似于：

http_parser_settings settings;
settings.on_url = my_url_callback;
settings.on_header_field = my_header_field_callback;
/* … */

http_parser *parser = malloc(sizeof(http_parser));
http_parser_init(parser, HTTP_REQUEST);
parser->data = my_socket;
当在套接字上接收到数据时，执行解析器并检查错误。

size_t len = 80*1024, nparsed;
char buf[len];
ssize_t recved;

recved = recv(fd, buf, len, 0);

if (recved < 0) {
/* Handle error. */
}

/* Start up / continue the parser.

  • Note we pass recved==0 to signal that EOF has been received.
  • /
    nparsed = http_parser_execute(parser, &settings, buf, recved);

if (parser->upgrade) {
/* handle new protocol /
} else if (nparsed != recved) {
/ Handle error. Usually just close the connection. */
}
http_parser需要知道流的结尾在哪里。例如，有时服务器发送响应时不使用Content-Length，并期望客户端在EOF之前使用输入（对于主体）。要告诉http_parser关于EOF，将0作为http_parser_execute()的第四个参数。在EOF期间仍然会遇到回调和错误，因此必须准备好接收它们。

标量值消息信息，如status_code、method和HTTP版本存储在解析器结构中。此数据只临时存储在http_parser中，并在每个新消息上重置。如果以后需要此信息，请在headers_complete回调期间将其从结构中复制出来。

解析器对请求和响应的transfer-encoding进行透明解码。也就是说，分块编码在被发送到on_body回调之前被解码。

升级的特殊问题
http_parser支持将连接升级到其他协议。一个越来越常见的例子是WebSocket协议，它发送一个请求

GET /demo HTTP/1.1
Upgrade: WebSocket
Connection: Upgrade
Host: example.com
Origin: http://example.com
WebSocket-Protocol: sample

后跟non-HTTP数据。

（有关WebSocketprotocol.的更多信息，请参阅RFC6455）

为了支持这一点，解析器将把它视为没有正文的普通HTTP消息，同时发出on_headers_complete和on_message_complete回调。但是http_parser_execute(）将在头的末尾停止解析并返回。

用户应该在http_parser_execute()返回后检查parser->upgrade是否已设置为1。Non-HTTP数据从缓冲区提供的偏移量开始，偏移量由http_parser_execute()返回。

Callbacks
在http_parser_execute()调用期间，http_parser_settings中设置的回调将被执行。解析器维护状态，从不向后看，因此不需要缓冲数据。如果需要保存某些数据以备以后使用，可以从回调中执行此操作。

有两种类型的回调：

notification typedef int (http_cb) (http_parser); Callbacks: on_message_begin, on_headers_complete, on_message_complete.
data typedef int (http_data_cb) (http_parser, const char *at, size_t length); Callbacks: (requests only) on_url, (common) on_header_field, on_header_value, on_body;
成功时回调必须返回0。返回non-zero值将向解析器指示错误，使其立即退出。

对于需要在回调中传递本地信息的情况，可以使用http_parser对象的data字段。这种情况的一个例子是使用线程处理套接字连接、解析请求，然后通过该套接字给出响应。通过实例化包含相关数据的thread-local结构（例如接受的套接字、为回调写入的分配内存等），解析器的回调能够以线程安全的方式在线程范围和回调范围之间通信数据。这允许http_parser在multi-threaded上下文中使用。

背景

wrk是团队开发的压测平台所使用的压测引擎，在使用上支持对固定域名服务的不同接口进行高并发压测，相较于其他压测工具，使用了同redis一样的AE事件驱动器, 通过I/O多路复用技术，使得一个线程可以无阻塞处理多个socket连接，从而达到使用更少的资源（thread）产生更大的并发（socket）的效果。

wrk对外提供一些回调函数，支持使用lua脚本，对请求前、请求中、请求后进行一些处理。这使得wrk不仅可以压测固定的url, 相同域名下的所有url都可以在一次压测中进行，达到全链路压测的效果。

但问题是，对于某些收购的项目，全链路压测中的需要压测到的接口，涉及了多个域名，并且多个域名的url请求之间还有数据传递，所以需要研究并修改wrk源码，使得在request回调里，能修改域名变量，以方便对域名+path的url做处理。

  2. redis AE实现

c & lua

lua与C的交互只能通过栈来执行。其交互形式也分两种，lua调用c函数，c解释执行lua代码。

Lua 使用一个 虚拟栈 来和 C 互传值。 栈上的的每个元素都是一个 Lua 值 （nil，数字，字符串，等等）。

无论何时 Lua 调用 C，被调用的函数都得到一个新的栈， 这个栈独立于 C 函数本身的栈，也独立于之前的 Lua 栈。 它里面包含了 Lua 传递给 C 函数的所有参数， 而 C 函数则把要返回的结果放入这个栈以返回给调用者

c语言语法

宏定义#ifndef……#define……#endif主要是为了避免头文件重复引用

其实“被重复引用”是指一个头文件在同一个cpp文件中被include了多次，这种错误常常是由于include嵌套造成的。比如：存在a.h文件#include “c.h”而此时b.cpp文件导入了#include “a.h” 和#include “c.h”此时就会造成c.h重复引用，这个时候，在重复引用的头文件中使用#ifndef……#define……#endif是非常有必要的。

首先，源代码在编译之前需要进行预处理（也就是预编译），预编译干什么呢？我所知道的就是进行头文件展开、宏定义替换等操作。

b.cpp包含a.h头文件的时候，就已经把A_H和C_H这两个宏添加到自己本文件的作用域内了，说明宏的作用范围只属于它的本身文件，所以b.cpp文件不能直接就对A_H和C_H宏进行判断。
